1. Purpose & scope

Serendib Gems is a gem business in Sri Lanka. Many gem businesses still work on paper and separate spreadsheets, so it is hard to know the real stock, the real profit, or where each stone is. This ERP puts everything into one connected system with six modules that share the same data, plus a login with roles and a dashboard. It is a prototype: it does not need every feature of a commercial ERP, but it must clearly show how business processes join together and how data flows between modules. It will be deployed to Microsoft Azure and be reachable by a public URL during the presentation.

2. Users & roles

  • Administrator — can see and do everything. Used for the demo and setup.
  • Inventory & Procurement staff — manage lots, suppliers, and purchases.
  • Workshop staff — manage cutting jobs.
  • Sales staff — manage customers, orders, invoices, and payments.
  • HR staff — manage employees.

For a prototype, roles mainly control which menu items and pages a user can open.

3. Environment & constraints

  • Client: any modern web browser on a laptop or phone.
  • Server: Node.js on Azure App Service. Database: Azure SQL Database.
  • Must deploy to Azure and be reachable by a public URL. Database must be Azure SQL (relational), as named in the brief.
  • One GitHub repository with regular commits from all members. The team writes JavaScript, so the stack is React and Node.

What you must know
  • Today the gem house runs on paper and disconnected files: unclear stock, and a risk of selling a stone twice.
  • The ERP fix is one shared record per stone plus live reports.
  • This is a clear digital transformation story for the report and slides.

Serendib Gems is a gem house in Sri Lanka. Sri Lanka is famous for gemstones, especially sapphires from the Ratnapura area. A gem house buys rough stones, gets them cut and polished, keeps them in stock, and sells finished stones to customers, both local and foreign.

The five main processes

  1. Buying rough stones from suppliers (miners, dealers, estates).
  2. Cutting and polishing. Skilled cutters turn a rough stone into a finished stone.
  3. Holding stock. Finished stones are kept safely, each with details and a lab certificate.
  4. Selling. Customers buy finished stones, by visiting or through the online shop.
  5. Handling money and staff. The business raises invoices, takes payments, manages staff.

Stakeholders

  • Owner or manager — wants real stock, real profit, and what is selling.
  • Procurement staff — buy rough stones and deal with suppliers.
  • Cutters — do the cutting work in the workshop.
  • Sales staff — deal with customers and orders.
  • Accounts staff — handle invoices and payments.
  • Suppliers & Customers — the two outside parties.

The problems today

  • Stock is not clear. The stock book is on paper, so nobody is sure what is really in stock.
  • Data is repeated and does not match across the purchase book, stock book, and sales bill.
  • Hard to track a stone. Is it still rough, being cut, reserved, or sold?
  • Slow reporting. Finding profit or the best selling gem type means adding paper by hand.
  • No link between parts. Buying, cutting, selling and accounts are separate.
  • Risk of selling the same stone twice, because there is no single shared record.

What the ERP gives

  • One shared record for each stone, so stock is always correct.
  • A single status field showing exactly where each stone is in its life.
  • Automatic links: a bought stone appears in stock at once, a sold stone drops stock at once.
  • Fast reports: the dashboard shows stock value, sales and low stock live.
  • No double selling: a stone is marked Reserved as soon as it is added to an order.
  • Better decisions, because the owner can see real numbers any time.

What you must know
  • We picked React + Node + Azure SQL over .NET to reuse the storefront and our JavaScript skills.
  • The API and the React app are one app, so there is one Azure deploy and fewer things to break.
  • Be ready to explain this choice in the viva.

React + Node/Express + Azure SQL, served as a single app.

For an assignment focused on Microsoft Azure, the obvious path is ASP.NET Core with Entity Framework, the most Azure native option. We chose React + Node/Express with Azure SQL instead. The reason: the team already built a polished React gem storefront whose components and design language we reuse for the ERP screens, and the team writes JavaScript, not C#. In a four day sprint, reusing the existing React work and staying in one language is worth more than the small extra Azure fit that .NET would give.

The Express server does two jobs. It serves the built React admin app, and it exposes the REST endpoints under /api. Because both come from the same server, the whole system deploys as one Azure App Service: no CORS, no second host, no server side rendering setup. This is the lowest risk way to still have a real backend and a real database, which protects the Azure Deployment and Database marks. We kept Azure SQL (not Cosmos or MongoDB) because the brief names Azure SQL Database directly, and the marks reward relational database functionality.

Considered and rejected

  • ASP.NET Core + EF Core + Azure SQL — most Azure native and best possible marks, but needs C# and a full UI rebuild, throwing away the React storefront.
  • Next.js + Prisma + Azure SQL — one clean codebase and reuses React, but Next.js SSR on Azure App Service has setup issues that can waste time in a short sprint.

Every gemstone is one Lot. A Lot has one status field. Three different modules change that same field during the normal business flow, so the modules are truly integrated, not just placed side by side.

Procurementbuys rough Inventoryrough in stock Cuttingrough → finished Salesreserve, then sell Billinginvoice + payment Dashboardlive totals
  1. Procurement buys a rough stone. The system creates a Lot with status In Stock (rough).
  2. Cutting runs a job. The status becomes In Cutting, then In Stock (finished).
  3. Sales adds the finished Lot to a customer order. The status becomes Reserved, then Sold.
  4. Billing creates an invoice for that order and records payment.
  5. The Dashboard reads all of this and shows live totals.

So one purchase in Procurement changes what Inventory shows, what Cutting can work on, what Sales can sell, what Billing charges, and what the Dashboard reports. This single chain is what we demonstrate live and draw on the "Interaction Between ERP Modules" slide.
